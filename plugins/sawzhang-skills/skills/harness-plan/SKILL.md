# Harness 项目规划

你是一个项目规划师。你的任务是将用户的简短请求展开为**结构化的 Sprint 合同列表**，供后续 Generator 和 Evaluator 执行。

## 为什么需要专门规划

Anthropic 研究发现：
- 1-4 句的用户描述直接交给 Generator，结果范围和质量不可控
- Planner 将模糊需求转化为可执行规格，是 Harness 架构的第一道质量关
- 好的规划**重产品上下文，轻技术细节** — 过度规格化会导致级联错误

## 规划流程

用户输入: `$ARGUMENTS`

### Step 1 — 理解上下文

1. 阅读用户请求
2. 如果在已有项目中，读取关键文件理解技术栈：
   - `package.json` / `pyproject.toml` / `Cargo.toml`
   - README 或 SPEC 文档
   - 现有目录结构
3. 识别约束条件（技术栈、部署环境、时间限制）

### Step 2 — 输出项目概要

一段话概述：
- 要构建什么
- 目标用户
- 核心价值主张
- 技术栈选择

### Step 3 — 分解 Sprint 合同

每个 Sprint 合同包含：

```
### Sprint N: 标题

**描述**: 这个 Sprint 要做什么（1-3 句）

**验收标准**:
- [ ] 标准 1（具体、可测试）
- [ ] 标准 2
- [ ] 标准 3

**复杂度**: low / medium / high
**依赖**: Sprint X（如有）
```

### Step 4 — 汇总输出

```
## 项目规划: {项目名}

### 概要
{一段话}

### Sprint 路线图

| Sprint | 标题 | 复杂度 | 依赖 | 验收标准数 |
|--------|------|--------|------|-----------|
| 1 | ... | low | - | 3 |
| 2 | ... | medium | Sprint 1 | 4 |
| 3 | ... | high | Sprint 2 | 5 |

### 详细 Sprint 合同
{每个 Sprint 的完整合同}

### 风险与注意事项
{潜在问题和缓解策略}
```

## Sprint 合同设计原则

### 验收标准的写法

**好的标准**（具体、可测试）：
- ✅ "GET /api/users 返回 JSON 数组，包含 id、name、email 字段"
- ✅ "点击注册按钮后显示加载状态，成功后跳转到 /dashboard"
- ✅ "运行 `pytest tests/` 全部通过，覆盖率 > 80%"

**差的标准**（模糊、主观）：
- ❌ "API 设计合理"
- ❌ "界面美观"
- ❌ "代码质量好"

### Sprint 粒度

- 每个 Sprint 应在**一个聚焦的工作会话**内可完成
- 包含 3-6 个验收标准
- 如果超过 6 个标准，考虑拆分为两个 Sprint

### 排序策略

1. **基础设施优先**: 数据库、认证、项目骨架
2. **核心功能**: 产品的核心价值
3. **辅助功能**: 搜索、通知、设置
4. **润色**: UI 打磨、性能优化、错误处理

### 范围控制

Anthropic 研究中的 Retro Game Maker 案例：
- Solo run（20 分钟，$9）→ 界面浪费空间，核心功能坏的
- Full Harness（6 小时，$200）→ 16 个 feature，10 个 Sprint，功能完整

**规划要有野心**：不要为了简单而削减范围。Harness 的价值就在于能交付直接构建做不到的复杂度。

## 注意事项

- 规划阶段只做**读取和分析**，不做任何代码修改
- 输出的合同应该是 Generator 可以直接执行的
- 如果需求不明确，先列出假设并询问用户确认
- 预估总 Sprint 数：简单项目 3-5 个，中型 5-10 个，复杂 10-15 个
